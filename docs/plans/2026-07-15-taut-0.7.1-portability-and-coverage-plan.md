# Taut 0.7.1 Portability and Coverage Recovery Plan

Date: 2026-07-15

Status: release recovery in progress. The initial 0.7.1 tags point to
`873c6af61013dd3bb9166da04b771c78e2b14ea4`; publication is blocked because
the first Windows MCP lane exposed portable-test path assumptions and a test
budget too short for 101 real SQLite commits on that runner. No 0.7.1 GitHub
Release exists. Portable test corrections and fresh exact-SHA evidence are
pending; tag recovery requires an explicit owner decision.

Class: 3, standard. The work changes reusable CI behavior and its documented
ownership under [DOM-5], [DOM-10], [DOM-11], and [DOM-15]. It does not change
the MCP product contract. The final coordinated 0.7.1 publication is an
explicitly requested routine release through the unchanged [TAUT-12.5]
helper, with no skipped gate, retag, or manual publication path.

Plan type: CI correction, portability proof, documentation closeout, and
coordinated patch release. No normative spec revision is proposed.

## Goal

Correct the stale MCP publication note, add one real macOS and one real Windows
SQLite-only MCP compatibility lane, recover coverage lost by incomplete xdist
worker collection, and publish all four packages as coordinated 0.7.1 releases
through the existing exact-SHA GitHub release path.

Keep the change narrow. Do not add a workflow-presence policy, a new smoke-test
taxonomy, a coverage framework, an arbitrary percentage floor, or product code
for portability failures that have not occurred.

## Requested Outcomes

- `docs/implementation/07-taut-mcp-architecture.md` no longer says that
  `taut-mcp` is unpublished. It records the first published MCP release,
  release SHA, canonical workflow evidence, and GitHub Release.
- `.github/workflows/test-mcp-extension.yml` runs its existing complete
  SQLite/PostgreSQL matrix on Ubuntu and, in one additional matrix job, runs
  the existing `not pg_only` suite on `macos-latest` and `windows-latest` at
  one representative Python version without a PostgreSQL service.
- The canonical root coverage owner collects direct root and Summon unit tests
  serially. Ordinary compatibility lanes remain parallel, and the intentional
  two-worker Summon process topology remains unchanged.
- The next canonical coverage report contains all four named shards and
  recovers the existing root/CLI execution that disappeared in 0.7.0.
- Changelog, implementation notes, repository map, plan index, and plan
  evidence agree with the final workflow and release state.
- Core, PG, Summon, and MCP are released together as 0.7.1 with the unchanged
  helper and all normal local, exact-SHA workflow, artifact, and release gates.

## Current Evidence and Diagnosis

The 0.7.0 aggregate reported 9,252 statements with 1,213 missed, or 86.89%.
The prior comparable run reported 8,001 statements with 736 missed, or 90.80%.
The new MCP shard was present and healthy: 1,223 statements with 99 missed,
or 91.91%. Adding MCP slightly improved the rate; it did not cause the drop.

Removing MCP from the 0.7.0 arithmetic leaves 8,029 statements with 1,114
missed. Existing core and Summon code therefore lost 378 covered statements.
Of those, 370 are concentrated in five unchanged files:

| File | Prior misses | 0.7.0 misses | Increase |
|------|-------------:|-------------:|---------:|
| `taut/_scripts.py` | 6 | 200 | 194 |
| `taut/commands/_summon_compat.py` | 3 | 69 | 66 |
| `taut/watcher.py` | 86 | 132 | 46 |

| `taut/commands/_registry.py` | 2 | 32 | 30 |

All tests passed. The root coverage log created two xdist workers, but the
uploaded root-unit artifact retained direct-test data from only one worker.
The prior artifact had the same latent one-worker defect; test scheduling
happened to place the high-coverage files on the retained worker. The new test
mix changed scheduling and exposed the incomplete collection.

The final combine also warned about one initialized but empty
`coverage.summon-process` child database. That file is not the cause: the lost
execution belongs to root direct-test modules, while the MCP and Summon
artifacts downloaded. Zero-byte or interrupted child files also existed in
the earlier healthy report. Do not pin coverage.py or introduce a coverage
combiner on this evidence.

Before implementation, re-read the plan against the current two workflows and
confirm each extra mechanism answers an observed failure. Before release,
re-read the full diff and canonical evidence with special attention to Windows
path identity, stdio behavior, coverage ownership, and exact-SHA release
selection. Do not proceed while a review finding or deviation row is
unresolved.
