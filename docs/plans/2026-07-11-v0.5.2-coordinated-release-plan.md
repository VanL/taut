## Goal

Bump `taut`, `taut-pg`, and `taut-summon` from 0.5.1 to 0.5.2, then publish
all three through the repository's GitHub-only batch release path. This moves
the two extensions off their failed 0.5.1 tags while preserving one tested
commit for all three 0.5.2 tags.

- All package metadata and first-party dependency floors read 0.5.2.
- The retained Summon lock resolves the local 0.5.2 core and Summon packages.
- The accidentally tracked PG lock is removed and ignored, preserving the
  documented no-retained-PG-lock boundary.
- One clean commit is the source of `v0.5.2`, `taut_pg/v0.5.2`, and
  `taut_summon/v0.5.2`.
- The repository release helper runs its complete local `all` gate without
  `--skip-checks`.
- All three GitHub Actions release gates succeed and create non-draft,
  non-prerelease GitHub Releases with the expected wheel and source archive.

## Context and Key Files

- `pyproject.toml` owns the core version and the root development dependency
  on `taut-summon`.
- `taut/_constants.py` owns the runtime core version string and must equal the
  root project version.
- `extensions/taut_pg/pyproject.toml` owns the PG package version and its exact
  current-core floor.
- `extensions/taut_summon/pyproject.toml` owns the Summon package version and
  its exact current-core floor.
- `extensions/taut_summon/uv.lock` is retained release state and must resolve
  the local core and Summon versions.
- `extensions/taut_pg/uv.lock` is not retained release state. Repository
  architecture requires it to remain absent; `.gitignore` prevents a local PG
  environment from silently reintroducing it into a release commit.
- `bin/release.py all` discovers every current version without an existing
  GitHub Release, runs the union of local release gates, builds fresh paired
  core/Summon artifacts, pushes the clean branch, and pushes all selected tags.
- The release workflows verify tag stability before publishing artifacts.

Comprehension gates:

1. Why must the version bump be committed before `bin/release.py all`? The
   batch path rejects a dirty worktree and treats current version files as its
   source of truth.
2. Why must all tags point to one commit? Core and Summon form a paired reactor
   boundary, and the installed-artifact canary must prove the exact wheels
   built from the tagged checkout.

## Invariants and Constraints

- GitHub Releases remain the publication boundary; do not upload to PyPI.
- Preserve tag namespaces: `vX.Y.Z`, `taut_pg/vX.Y.Z`, and
  `taut_summon/vX.Y.Z`.
- Do not move or delete any existing 0.5.0 or 0.5.1 release/tag.
- Do not use `--skip-checks`; local LLM, external harness, PG, lint, typing,
  build, lock, and installed-artifact gates remain mandatory.
- Do not force-push. The new 0.5.2 tags must not exist before this release.
- Keep `taut>=0.5.2` exact in both extensions and
  `taut-summon>=0.5.2` in the root development extra.
- Retain and refresh only the Summon lock. Do not generate or commit a PG lock;
  PG dependency compatibility is proved from its manifest by the fresh paired
  artifact verifier.
- Do not change runtime behavior, package dependencies other than the
  first-party version floors, workflow contracts, or release tooling.
- The tests must exercise real Docker/Postgres, process, external harness, and
  local LLM boundaries as selected by `bin/release.py`; do not mock them.

## Rollout and Rollback

Rollout is ordered by the existing helper: commit the synchronized version
state, pass local gates, push `main`, then push the three tags. GitHub Actions
publishes only after each tag-stability and package gate succeeds. Core may
publish before Summon only as its immediate dependency; the coordinated
release is not complete until all three releases and artifacts exist.

Before tag push, rollback is a normal revert of the version commit. After a
tag creates a public GitHub Release, do not rewrite it silently. A failed gate
may be rerun only when the tag still points at the tested commit; any code fix
requires a new patch version unless the user explicitly authorizes a retag
before publication.
